# 運用判断ログ（Decisions）

「**いつ・何を・なぜそう決めたか**」を蓄積するページ。
3ヶ月後・1年後に「なぜ今この設定なのか」を思い出すために、特に**判断が分かれた論点**や**複数案から選んだもの**を記録する。

> 単なる機能追加（迷う余地のない実装）は記録不要。**判断・選択・トレードオフ**があったものを残す。

## 2026-04-30 上智大学渋谷10/24 が毎回新規企画参加扱いされた現象は一時的事象と判定、修正不要

**決定**: コードに恒常的なバグはないと判定し、**修正は行わない**。一時デバッグログも削除して通常運用に戻す。

**背景**:
- 2026-04-28 の 17:44/18:44/19:29/20:29 の autoImportCSV ログで、**同じ「上智大学 渋谷10/24: ○」が毎回「新規企画参加」として検出され続けた**
- スナップショットには該当 id がある（`縦持ち_自動` シート確認済み）にもかかわらず diff が立つ謎現象
- 4/28夜にデバッグログ仕掛け → 4/30朝の trigger 実行ログを取得して分析

**4/30 9:44 ログでの確認結果**:
- `diffMsg = "(差分なし)"` ← 新規検出なし
- 渋谷10/24 含み id 81件すべて `snap.byId に同一id存在? true`
- 上智大学（10156A）の id 4件（水戸6/1・長野6/10・松本6/11・渋谷10/24）すべて snap と完全一致
- hex 値も期待通り（不可視文字混入なし）

**代替案と却下理由**:
- コード修正（snapshot や id 生成のロジック改修） → 再現しないものを直しても意味がない、副作用リスクのみ
- デバッグログ常設 → 通常運用のログがノイズで埋もれる
- 「毎回 diff 出ても無視」運用 → 真のデータ変動を見落とすリスクあり

**推定原因**:
- 4/28 当時、Phase 2（縦持ち_自動 への書込み）が何らかの理由で完了していなかった、もしくは書込み直前のスナップショット↔新データ生成↔書込みのタイミングで一時的に齟齬が発生
- 一過性のため再発までは原因確定不能

**影響**: なし（業務データには影響なし。取込ログのノイズのみ）

**関連**:
- コミット `41c9d87` (LuggageWebApp) — デバッグログ追加（4/28夜）
- コミット `dd5d211` (LuggageWebApp) — デバッグログ削除（4/30朝、調査完了後）
- 関連シナリオ: [シナリオ9: トラブル対応](scenarios.md) — 「再発したら再仕込み調査」の選択肢あり

## 2026-04-27 SS差異検知の空白正規化を A案統一に変更

**決定**: 全フィールド（部署・担当者・メール）で **空白完全除去** で比較する（A案）

**背景**:
- 4/24 に B案（部署・担当者は意図的な空白区切りを保持・メールのみ全空白除去）で導入したが、**営業の入力で氏名間のスペース有無が揺れる**ため差異検知が発火しすぎ、確認作業が増えていた
- 読み込み時間も長いので、件数が多いと時間ロスが激しい

**代替案と却下理由**:
- B案維持（現状） → ノイズが多く運用負荷が大きい
- 「空白のみの差は自動で確認済み」 → 中規模実装が必要・ロジック複雑化
- 確認済み一括ボタン → ユーザーが望む時だけ動かせるが、結局判断負荷が残る

**影響**: 「山田太郎」と「山田 太郎」が同値扱いになる。営業の入力規約が空白ベースで揺れている現状ではこの「区別の損失」は無害

**関連**:
- コミット `2bda0a6` (LuggageWebApp)
- `Opview.js` `normalizeForDiff_`
- 関連ガイド: [参加状況ビューア > 営業SS差異の確認・反映](guide-opview.md)

## 2026-04-27 配付物管理 会場別封入クロス表で栄美通信品目を本社管轄のみ包含

**決定**: `getDistVenueMatrix` で栄美通信CL品目を **東京本社管轄会場のみ包含**（支社管轄は従来通り除外）

**背景**:
- 「ルート別と会場別で品目数が違う」というユーザー指摘から差異3点が判明（セクション・栄美通信除外・部数>0）
- 栄美通信品目は本社では実体があるが、支社では除外したい

**代替案と却下理由**:
- 全会場で栄美通信を含める → 支社業務に不要なノイズが増える
- 全除外維持 → 本社業務で必要な品目が落ちる

**影響**: 東京本社管轄会場の行のみ栄美通信品目列が値付きで表示される

**関連**: コミット `576647d`、`distributionHandler.js`、[配付物管理 > 会場別タブ](guide-distribution.md)

## 2026-04-24 担当者情報変更チェックは手動運用を維持

**決定**: 荷物登録フォームの「担当者情報変更」チェックボックスは、氏名・メアド変更を検知しても **自動でONにしない**

**背景**:

**関連**: `Index.html`、[荷物登録フォーム > 担当者情報の変更チェックボックス](guide-form.md)

## 2026-04-24 venueMatch: 7/10 難波御堂筋ホールの city を `難波` に統一

**決定**: スクレイピング側のデフォルト挙動（住所由来で `city="大阪"`）ではなく、**業務呼称（企画設定の「開催都市」）に寄せて `city="難波"`** とする。エイリアス `難波→大阪` は撤回

**背景**:
- 6/6 にハービスホール（梅田）が `city="大阪"` で別会場として存在
- 7/10「難波」と6/6「大阪」を同一視するとデータ衝突リスク

**代替案と却下理由**:
- エイリアス `難波→大阪` を残す → 別会場が同一キーに衝突。同日複数会場では誤マッチ
- スクレイピング側のロジックで「住所由来の都道府県補完」を維持 → 業務呼称と乖離

**影響**: ハマーン殿側のスクレイピング再生成時に `city="大阪"` に戻ったら、こちら側で再修正が必要（恒久対策はスクレイピング側のロジック改善依頼）

**関連**: コミット `7dad467` `2deec03`、`venueMatch.js`、`venues.json`

## 2026-04-24 集荷案内自動下書き発動を 30〜37日前に変更（旧 14〜21日前）

**決定**: `autoCreateGuidanceDrafts` の対象を **期限30〜37日後**（締切1ヶ月前発動）に変更

**背景**:
- 旧設定（14〜21日後）では参加校への通知が遅すぎ、フォーム回答までの猶予が短い
- 1ヶ月前に通知すれば余裕を持って準備できる

**代替案と却下理由**:
- 21〜28日前 → 月末月初の処理が前期と重複しがち
- 45日以上前 → 早すぎて参加校の意識から流れる

**関連**: コミット `63b9bf8`、`guidanceConfig.js`

## 2026-04-24 実施ガイド導線をフォームから廃止しリマインドメール末尾に集約

**決定**: 荷物登録フォーム冒頭の「📘 実施ガイド」ボタンを廃止し、リマインドメール本文末尾の緑カードに導線を一元化

**背景**:
- iframe 配下からの meta refresh が一部環境（Chrome マルチアカウント・企業プロキシ）でブロックされる問題
- 中継ページ jg.html を能動クリック型ランディングに変えても効果薄く、根本解決困難

**代替案と却下理由**:
- jg.html を能動クリック型に変える → 一部環境では結局 iframe 制約で開けない
- フォームに警告だけ表示 → 改善にならず混乱を招く

**影響**: 顧客は実施ガイドにメール経由でしか辿れなくなるが、これは元々メール起点運用なので問題なし

**関連**: コミット `91eeda0` `97d0dbd`、`Index.html`、`Remindermail.js`

## 2026-04-23 ダッシュボード単校送信に block/warn 2段階診断を導入

**決定**: ダッシュボードからの単校リマインド送信で、致命的不整合（メール未設定等）は **block**（送信不可）、警告（全会場終了済み等）は **warn**（confirmで突破可能）として扱う

**背景**:
- 自動抽出ロジックから単校強制送信を派生させる際、すべて同列に扱うと運用事故リスクと柔軟性が両立しない

**代替案と却下理由**:
- 全部送信可能（無制限） → メール未設定校に空アドレスで送信する事故リスク
- すべてブロック → 「終了済みだが念のため通知」等の柔軟運用ができない

**影響**: warn は人間判断で突破できるので運用柔軟性を保ちつつ、致命的事故は防げる

**関連**: コミット `41ed496` `2e76014`、`Remindermail.js` `diagnoseReminderForSchool_`

## 2026-04-23 集荷案内/リマインドメールで終了済み会場をフィルタ

**決定**: 開催日が前日以前の会場（終了済みイベント）は集荷案内・リマインド対象から除外

**背景**:
- 物流期限（route 単位）とイベント開催日（event 単位）は別次元
- ルート単位の期限フィルタだけでは「終了済みイベント」を除外できず、自動化処理が終了後も動き続ける事故が起きていた

**代替案と却下理由**:
- 期限フィルタのみで対応 → 上記の通り終了イベントが残る
- 当日も除外 → 当日朝のリマインドが必要なケースに対応できない

**影響**: 縦持ち_自動 N列の開催日を見て前日以前を除外。開催日不明は残す（過剰除外回避）

**関連**: コミット `87cf1fc` `c92278e`、`Guidancemail.js` / `Remindermail.js` `parseEventDateMMDD_`

## 2026-04-17 法人グルーピングのソートキーは「代表校が参加しているか」に依存させない

**決定**: 法人グルーピングのソートキー構築で、代表校が当該ルート不参加でも子校が法人グループから外れないよう、学校基本情報など全校マスタからの `kanaByCode` フォールバックを用意する

**背景**:
- 参加校マップ（`schoolMap`）だけを参照すると、代表校が当該ルート不参加の場合に `sortKana` が undefined になり子校が法人グループから外れていた

**代替案と却下理由**:
- 代表校を必ず参加扱いする → 実態と乖離し他のロジックを壊す
- 代表校不参加時は法人グループを解体 → 業務上「同じ法人として扱いたい」要件と矛盾

**影響**: `isRep` フラグを明示的に付与し、同グループ内の順序（代表校先頭→子校フリガナ順）を保証

## 2026-04-16 下書きメールは実送信まで送信済み扱いしない

**決定**: Gmail下書きを作成しても M列ステータスを「送信済み」扱いに更新しない。**実送信（成功ログ）時のみ**「前期済 MM/DD」に更新する

**背景**:
- Gmail下書きはユーザーが削除可能で、下書き作成時にM列を更新すると削除時に「送信済み」のまま残り送信漏れが発生する

**代替案と却下理由**:
- 下書き作成時に M列を「下書き MM/DD」と区別表示する → 実装済み（これは保持）。ただし「送信済み」とは別表記にして、実送信時に上書きする設計
- 下書き削除を不可に制限する → ユーザー体験を悪化させる

**影響**: 下書き削除しても再実行で再作成される運用に。`checkAllSentDrafts` トリガーで送信済みを検知して自動更新

**関連**: `checkSentDrafts.js`、`guidanceMail.js`、`Remindermail.js`

## 過去決定の整理ルール

- **新しいものを上に**（時系列降順）
- **判断が分かれた論点のみ記録**。実装上自明な機能追加は記録不要
- **代替案と却下理由を必ず書く**：将来「やり直したい」と思った時の判断材料になる
- 関連コミット・ファイルへのリンクで詳細は git/code に委ねる
- **撤回・変更があれば**、新しい決定として上に追記し、過去決定の末尾に「→ YYYY-MM-DD 撤回。(該当エントリ参照)」を追記
