# シナリオ別ガイド（業務目線の逆引き）

「**この業務をやりたい時にどの画面・どのトリガーを触るか**」を逆引きで把握するためのページです。
画面ごとの詳細は各 [運用ガイド](README.md) を参照してください。

## シナリオ1: 集荷案内を出して回答を集める

### 流れ

```
①月初準備     ②自動下書き    ③下書き確認・送信   ④回答受付
企画設定/物流  毎朝 7-8時     Gmail下書き         荷物登録フォーム
設定の確認  →  自動作成     → 内容確認 → 送信   → 参加校が回答
```

### 詳しい手順

1. **月初準備** — [企画設定／物流設定](guide-opview.md) で当月分の会場・送付期限が登録されているか確認
2. **自動下書き作成（自動）** — 毎朝7-8時、`autoCreateGuidanceDrafts` が **期限30〜37日後** の校を抽出して Gmail下書きを自動作成。`chikoshima@eibi.co.jp` に件数通知メールが届く
3. **下書き確認・送信** — Gmail下書きフォルダで内容を目視 → 問題なければ送信。1校ずつ送ってもよいし、ダッシュボードから全選択してもよい
   - 不要な下書きは削除可能（再実行で再作成される）
   - 緊急で1校だけ追加で送りたい場合は [参加状況ビューア > ダッシュボード](guide-opview.md) の「集荷 即送信」ボタン
4. **送信状態の反映** — 送信完了すると、自動で `集荷メール` 列が `前期済 MM/DD` に更新される。`checkAllSentDrafts`（1日2回・9時/18時）が下書き→送信済みを検知して反映

### 関連ページ

- [集荷案内・リマインドメール送信](guide-mail.md) — 画面操作の詳細
- [参加状況ビューア > ダッシュボード](guide-opview.md) — 単校送信ボタン

### 関連シート

- 学校基本情報（M列=集荷メール状態）
- 送信ログ
- 取込ログ

## シナリオ2: 未回答校にリマインドする

### 流れ

```
①自動抽出       ②自動下書き      ③送信
リマインド対象   毎朝 7-8時       Gmail下書きフォルダで確認 → 送信
を抽出       →  下書き作成     → （または1校だけ即送信）
```

### 対象抽出条件

- 集荷案内メール送信済み
- フォーム未回答 OR フォーム回答済みだが倉庫送付なのに未入庫
- 送付期限の3日前以内 OR 期限超過
- 開催日が翌日以降（**終了済み会場は自動スキップ**）

### 詳しい手順

1. **自動下書き作成（自動）** — 毎朝7-8時、`autoCreateReminderDrafts` が対象を抽出して下書き作成
2. **下書き確認・送信** — Gmail下書きフォルダで確認後、送信
3. **緊急の単校送信** — [ダッシュボード](guide-opview.md) の「ﾘﾏｲﾝﾄﾞ 即送信」ボタン（事前診断 block/warn 付き）

### 関連ページ

- [集荷案内・リマインドメール送信](guide-mail.md)

## シナリオ3: 届いた荷物を入庫する

### 流れ

```
①QRスキャン or 検索 → ②個口入力 → ③保存 → ④集荷確認タブで反映
```

### 大量入庫のとき

[入庫記録](guide-receiving.md) の **QRスキャンタブ** が最速：

1. 「カメラ起動」 → 荷札のQRコードをカメラに向ける
2. 学校名・ルート・予定個口が表示される
3. 入庫個口数を入力 → 「入庫登録」
4. 続けて次の荷札へ

### 1校ずつ慎重に入庫

[入庫記録 > 単校タブ](guide-receiving.md) で学校検索 → ルート選択 → 個口入力。

### 法人で送られてきた

[入庫記録 > 法人一括タブ](guide-receiving.md) で法人検索 → 配下校×ルートの一覧が出るので、各セルに入力 → 「一括登録」。

### 入庫ミスの修正

[入庫記録 > 履歴タブ](guide-receiving.md) で対象行を「編集」 or 「削除」。最大50件・入庫日新しい順。

### 関連ページ

- [入庫記録](guide-receiving.md)
- [参加状況ビューア > 集荷確認・仕分表](guide-opview.md) — 入庫状況の確認

### 関連シート

- 入庫管理（入庫個口列）

## シナリオ4: 配付物（資料）を手配する

### 流れ

```
①xlsx取込       ②品目詳細で調整   ③ルート別/会場別で確認   ④印刷・指示書化
顧客から      　ルートごとの    　現場用の確認資料を       倉庫スタッフへ
申込書を取り込む  部数を補正     　PDFで生成                 配布
```

### 詳しい手順

1. **xlsx取込** — [配付物管理 > xlsx取込タブ](guide-distribution.md) で顧客の申込書 Excel をアップロード
2. **品目詳細モーダルで調整** — 品目一覧の行クリック → モーダルが開く → 各会場の倉庫部数・事業課部数を入力
   - 持ち回り（チェックボックス）で「次の会場に持ち回し」を表現
3. **ルート別タブで確認** — 物流ルート単位で会場×品目の横展開表を生成。倉庫の発送作業確認用
4. **会場別タブで俯瞰** — 期全体の封入クロス表で「どの会場にどの品目が入るか」を一覧
5. **印刷** — 各タブの「印刷」ボタンで A4印刷可能。指示書として現場に配布

### 関連ページ

- [配付物管理](guide-distribution.md)

### 関連シート

- 配布物マスタ（品目情報）
- 配布先明細（会場別の部数）

## シナリオ5: リモート参加（Zoom配信）会場を運営する

### 流れ

```
①Webアカウント割当 → ②機材手配 → ③ブースラベル印刷 → ④URL連絡
```

### 詳しい手順

1. **Zoomブースの割当** — [参加状況ビューア > リモート参加 > ブース管理サブタブ](guide-opview.md) で `アカウント` 行に「割当校」をプルダウンから選択
2. **Tamaya機材の発注** — 同タブの `Tamaya機材列` で見積依頼〜発注No.〜返却までの日付を記録
3. **ブースラベルPDFを生成** — 各行の「ラベル」列の「生成」ボタン → 学校名・会場名と日付・ZoomID・ZoomPass・機材No.・レンタル終了が印字された A4横2×4の同一ラベル8枚PDFがダウンロード
4. **URL連絡** — `URL連絡` 列を `未→済` に更新

### 関連ページ

- [参加状況ビューア > リモート参加](guide-opview.md)

### 関連シート

- {期}Webアカウント管理

## シナリオ6: 年次更新（前期 → 後期 等の切替時）

### やること

1. **企画設定の更新** — 後期会場・日程・ルートを追加
2. **物流設定の更新** — 後期分の送付期限を入力
3. **持ち回りチェックボックスの再設定** — 会場順が変わった配付物詳細で持ち回りチェックを入れ直す（年次更新時の注意点）
4. **venues.json の追加スクレイピング依頼** — 後期実施ガイドPDF公開後、スクレイピング担当に新版生成を依頼 → eibi-form/docs/data/venues.json を更新 → GAS で `clearVenueCache()` 実行
5. **ペア校マスタの確認** — 短大データを大学にフォールバックするペアが新期でも維持されているか
6. **集荷案内/リマインドの「期」フィルタ動作確認** — 新しい期で正しく対象抽出されるか

### 関連ページ

- [配付物管理](guide-distribution.md) — 持ち回りチェックボックス
- [システム構成](admin-system.md) — venues.json 連携・ペア校マスタ

## シナリオ7: 担当者情報の変更があった

### 流れ（フォーム経由）

参加校が [荷物登録フォーム](guide-form.md) で「担当者情報変更」チェックを入れて送信
↓
`変更依頼` シートに記録
↓
**毎日 8:00 / 17:00** の `runCLChangeNotification` で営業SS差異と合わせて通知メール送信
↓
運営側で内容確認 → 学校基本情報を手動で更新 or [ダッシュボード](guide-opview.md) で編集

### 流れ（営業SS経由）

営業部が営業SSで担当者情報を更新
↓
`runCLChangeNotification` が学校基本情報との差異を検知して通知
↓
[ダッシュボード](guide-opview.md) の編集モーダルで「反映」ボタンを押して学校基本情報に上書き、または「確認済み」で現状維持

### 関連ページ

- [荷物登録フォーム > 担当者情報の変更チェックボックス](guide-form.md)
- [参加状況ビューア > 営業SS差異の確認・反映](guide-opview.md)

## シナリオ9: トラブル対応

| 症状 | 原因の見当 | 対処 |
|------|-----------|------|
| 画面が古い情報のまま | キャッシュが残っている | スプレッドシートのメニュー「システム」→「キャッシュクリア」 |
| メールが送られない | 自動下書きが土日スキップ | 平日まで待つ／必要なら手動で「集荷 即送信」 |
| ダッシュボード SS差異の赤バッジが消えない | 営業SSと学校基本情報に空白だけの差異 | 2026-04-24以降は空白正規化で同値扱い。それでも残るなら値を再保存 |
| QRが認識しない | テープ被覆 or 印刷品質 or カメラ照明 | 荷札を貼り直す／検索タブから手入力 |
| `府京都` が未ヒット | venues.json スクレイピングバグ | venueMatch.js の `CITY_SCRAPING_FIXES_` で自動補正済み（手動対処不要） |
| 入庫管理に空プレースホルダ行 | 倉庫側が予定校マーカーで挿入 | 2026-04-22 修正済み（フォールバック判定は通る） |

## シナリオ10: ナレッジ同期事故から復旧する

`project_knowledge.md` の Claude.ai 同期で **古いスナップショットから派生**してしまい、過去追記が消失するケースがある（2026-04-28 に発生）。発生メカニズムと復旧手順を残す。

### 発生メカニズム

Claude.ai 側でプロジェクト知識（プロジェクトに直接添付されているファイル）と、その時の会話で添付したファイルが混在し、**新しい添付より古いプロジェクト知識を優先してしまう**ことがある。あるいは過去会話で参照した版がコンテキストに残り、古い版に引っ張られる。結果、最近の追記（時系列マーカー・新規ファイル行・新しい設計原則 等）が静かに消える。

### 予防（毎回の同期サイクル）

Claude.ai に作業まとめを渡す時、添付指示文に必ず以下を明記する：

> 「添付した `project_knowledge.md`（コミットハッシュ `XXXXXX`、最終更新 YYYY/MM/DD）を絶対的なベースとしてください。Claude.ai 内のプロジェクト知識や過去会話の記憶ではなく、この添付ファイルだけを参照源とすること。」

それでも事故は起こりうるので、**毎回の差分検証を必須化する**。

### 検知（出力受領時）

新版 `project_knowledge.md` を上書き保存する**前**に必ず確認：

```bash
# 1. ヘッダーの最終更新日付・サマリー文言が想定通りか目視
head -3 ~/clasp-projects/LuggageWebApp/project_knowledge.md

# 2. 差分stat
git diff project_knowledge.md --stat

# 3. 削除行を全件確認（純粋な追記なら削除はゼロのはず）
git diff project_knowledge.md | grep "^-" | grep -v "^---"
```

特に注視すべきセクション：
- **§9 Wiki テーブル**：追記累積エリア。scenarios.md / calendar.md / decisions.md など過去新設行が消えやすい
- **§8 設計原則**：累積エリア。過去追加の原則が消えやすい
- **§2.3 GASスクリプト一覧**：各ファイル行の末尾マーカーが消えやすい
- **時系列マーカー**：`4/27更新` `4/28追加` 等の日付ラベルが落ちていないか

### 復旧（消失検知時）

#### A案 — 上書きを保留して Claude.ai に再依頼
- 「コミット `XXXXXX` をベースに、scenarios.md / calendar.md 行と§8の○○原則が消えています、修正してください」
- 完璧な版が手に入るが往復1サイクル消費

#### B案 — 上書き＋私（Claude Code）が消えた行を手動復元（推奨）
- 4/28 に発生したケースもこのパターンで処理
- 該当行は git の過去コミットから復元できる：

```bash
# 該当時点（消失前）のコミットハッシュ確認
git log --oneline -- project_knowledge.md | head -10

# 過去版から該当行を抜き出す
git show <hash>:project_knowledge.md | grep -n "<消失したキーワード>"

# 復元先にコピペ（Edit ツール or sed 等）
```

#### C案 — 何もしない
- 4/28 分等の新規追記を反映しない選択。データ消失リスクは回避するが追記の取り込みが遅れる

### 復旧後

- 復旧コミットには **「ナレッジ同期事故からの手動復旧」** と明記して履歴に残す（次回参照しやすくする）
- 消失パターンを `decisions.md` に記録するかは判断次第（ナレッジ運用の判断ログとしての価値はある）

### 関連事故

- 2026-04-24 — venueMatch 関連で同様の事故（4/22 以前のスナップショット派生）。`ee92230` で手動マージ復旧
- 2026-04-28 — 4/27 のWiki更新マーカー・scenarios.md / calendar.md 行・§8 2原則が消失。`e644be2` で手動復旧
